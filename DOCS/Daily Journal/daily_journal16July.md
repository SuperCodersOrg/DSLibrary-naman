## Section 1 — Specific Bug

### Bug 1
**Issue:** While shifting the DSLibrary project from the Windows toolchain to WSL (Ubuntu), CMake failed to configure the project because the existing build directory had been generated using a different build generator.

**Observation:** I learned that CMake stores generator-specific information inside `CMakeCache.txt`. Reusing the same build directory with another generator leads to configuration conflicts, making it necessary to create a fresh build directory or remove the existing cache before reconfiguring the project.

## Section 2 — Failed Attempt

Initially, I attempted to reuse the existing build directory that had been created while working on Windows. Since the directory already contained cache files generated by a different toolchain, CMake repeatedly produced generator mismatch errors.

After several unsuccessful build attempts, I removed the old build configuration and regenerated the project using the Linux GCC toolchain inside WSL. During compilation, I also encountered a missing `LinkedList.tpp` file due to an incorrect include path. After correcting the template implementation reference and rebuilding the project, the compilation completed successfully.

## Section 5 — Learning Reflection

Today's biggest learning was understanding how build systems and development environments affect the software development process. Earlier, I believed that changing from one compiler or operating system to another only required rebuilding the project. Through practical experience, I learned that CMake maintains generator-specific configuration inside cache files, making it essential to regenerate the build configuration whenever the development environment changes.

I also gained a better understanding of how template implementation files are organized in C++. Since template definitions must be visible during compilation, incorrect or missing `.tpp` includes immediately lead to compilation failures. Resolving these issues improved my understanding of template-based library design.

Finally, working inside WSL introduced me to a standard Linux development workflow. I learned how to configure projects using GCC and CMake, troubleshoot build errors systematically, and execute compiled binaries in a Linux environment. This experience gave me greater confidence in using professional development tools and established a more reliable workflow for future implementation and testing.
